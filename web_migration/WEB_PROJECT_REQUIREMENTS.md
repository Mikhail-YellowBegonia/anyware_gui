# WEB 项目需求草案（修订版）

## 1. 项目概况
- 项目名称：谷雨（Koyuu）
- 项目定位：轻量、可扩展、风格化的交互式 UI 框架（Web 端）
- 背景：从本地 Pygame GUI 框架迁移到 Web，目标是更轻、更稳、更易被 AI 与开发者理解
- 目标用户：
  - 框架作者本人（第一用户）
  - 基于框架做二次开发的开发者（第二用户）
- 项目组成：
  - 前端框架（核心）
  - 后端示例程序（通信与数据）
  - 示例应用（验证功能与风格）

## 2. 团队与能力画像（用于约束方案）
- 单人开发
- Python 基础较好
- HTML/CSS/JS 基础较浅
- 具备一定设计与美术能力
- AI 编程开放，但强调可控性、可维护性、可靠性

## 3. 产品愿景与成功标准

### 3.1 愿景
成为“构建特殊风格控制界面”的一个实用选项：上手快、风格明确、工程可控。

### 3.2 近期目标（MVP）
- 先完成可运行、可复用的最小框架骨架
- 用一个简化验证应用证明：
  - 通信链路稳定
  - 核心组件可用
  - 风格表达成立

### 3.3 成功标准
- 框架可独立使用（不是一次性 demo）
- 文档清晰，能指导非作者使用
- 至少 1 个完整示例 + 若干最小示例
- AI 能稳定生成与修改框架代码（低歧义）

## 4. 架构原则（关键决策）
- Web 标准优先：HTML/CSS/DOM/SVG/Canvas 优先于自定义渲染协议
- 先减法后加法：先做最小闭环，再扩展高级能力
- AI 友好优先：避免自定义 DSL、私有坐标体系、过深魔法层
- 组件语义保留、实现重写：保留“组件是什么”，不继承旧渲染实现
- 业务与渲染解耦：通信/状态/模拟逻辑尽量独立于 UI 层

## 5. 需求分级（MoSCoW）

### 5.1 Must（MVP 必须有）
- 渲染与界面
  - 标准 HTML/CSS 渲染
  - SVG 与 Canvas 2D 支持（用于仪表/图形组件）
  - CSS 动画与过渡
- 组件体系
  - 明确的组件接口规范（props/events/lifecycle）
  - 一组核心组件（按钮、面板、文本、基础仪表）
- 响应式与通信
  - 实时通信链路（WebSocket）
  - 前后端基础协议定义（消息类型、错误处理、重连策略）
- 布局与样式
  - 采用现代 Web 布局（Flex/Grid）
  - 支持局部绝对定位（在必要组件内）
  - 全局主题 + 组件级样式覆盖
- 工程化
  - 开发热更新
  - 生产构建
  - 最小脚手架（创建项目、运行 demo）

### 5.2 Should（V1 建议有）
- 多窗口/多屏幕布局管理（先从单页多区域开始）
- 更完整组件库（表单、列表、图表容器、状态提示）
- 插件机制（数据源扩展、生命周期钩子）
- 内置通信适配层（WebSocket 抽象 + Mock 通道）

### 5.3 Could（后续探索）
- 基础轻量 3D 能力（建议延后，避免早期复杂度飙升）
- 复杂嵌入式部署优化
- 可视化布局编辑器

### 5.4 Won’t（当前阶段不做）
- 复刻 Pygame 时代的完整 DSL 编译链
- 自定义全局坐标系统替代 DOM 布局
- 追求“全能型”组件库

## 6. 非功能需求（可量化）
- 性能
  - 常规页面交互流畅，目标 60 FPS
  - 高频动态图表场景不低于 30 FPS
- 资源占用
  - 浏览器内存占用在可控范围（以验证应用为基准持续监测）
- 稳定性
  - WebSocket 异常具备重连与降级策略
  - 核心模块需有基础自动化测试
- 可维护性
  - 统一代码规范、目录结构、命名规范
  - 新人可在文档指导下完成本地运行与简单扩展
- AI 友好
  - 需求、接口、组件规范文档化
  - 减少隐式行为与“约定大于文档”的情况

## 7. 技术栈建议（结合当前能力）

### 7.1 推荐方案（优先）
- 前端：React + Vite + TypeScript
- 样式：Tailwind CSS + CSS Variables（主题化）
- 组件基础：Radix/Headless 思路 + 自定义风格封装
- 状态管理：Zustand（轻量，学习成本低）
- 通信：原生 WebSocket（必要时封装重连策略）
- 后端示例：FastAPI（贴合 Python 能力）
- 测试：Vitest + Playwright（核心流程 + 回归）

推荐理由：
- AI 训练语料多，生成质量稳定
- 社区资料丰富，排障成本低
- 与单人开发节奏匹配

## 8. 迁移资产建议

### 8.1 可复用
- 业务模拟逻辑（如 reactor_sim）
- 后端示例与通信逻辑
- LLM 中间件思路（解析、分发、缓冲）
- 风格方向与视觉素材

### 8.2 需要重写
- 渲染内核（Pygame/OpenGL 相关）
- 旧组件实现（保留语义，重写 Web 版本）
- 布局 DSL 与坐标系统（以 Web 原生替代）

## 9. 风格与设计原则
- 核心理念：复古 = 轻量
- 开发顺序：功能优先，美术后置打磨
- 验收标准：最终版本对视觉质量保持高要求
- 风格关键词：
  - 主风格：复古未来主义、磁带未来主义、NASA 朋克
  - 次风格：Swiss Style（结构与秩序）
  - 视觉元素：无抗锯齿、矢量、网格、留白、柔光、发光

## 10. 风险与应对
- 风险：前端经验不足导致实现不稳定
  - 应对：优先采用主流栈 + 小步迭代 + AI 配合 + 明确代码规范
- 风险：早期目标过大导致延期
  - 应对：严格执行 Must/Should/Could 分层
- 风险：风格实现过早复杂化
  - 应对：先做结构与功能，风格效果分阶段叠加

## 11. 里程碑建议（可调整）
- M1：完成项目骨架与通信闭环（前端 + 后端 demo）
- M2：完成核心组件与基础主题系统
- M3：完成验证应用 + 文档初版 + 基础测试
- M4：评估扩展项（插件机制/多窗口/轻量 3D）
